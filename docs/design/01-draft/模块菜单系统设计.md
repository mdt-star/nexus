# 模块菜单系统设计

## 一、设计背景

设计一个轻量化的模块菜单系统，支持第三方模块通过 JSON 配置声明权限和菜单，后端自动解析落库到 `permissions` 和 `menus` 表。用户可从菜单池拖拽绑定到个人桌面，实现高度自定义的工作空间。

---

## 二、核心设计理念

| 理念 | 说明 |
|------|------|
| 权限与菜单分离 | permissions 负责权限树结构，menus 负责菜单展示数据 |
| 配置轻量化 | 自动推断字段，减少冗余配置 |
| 用户自定义 | menus 仅发布，不直接展示；用户从菜单池拖拽到桌面 |
| 多国语言 | name 通过语言包匹配 tag，数据库只存结构 |

---

## 三、表结构设计

### 3.1 permissions 表

权限节点字典表，仅存储层级结构。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint, PK | 主键 |
| parent_id | bigint, nullable | 父级 id，null 为顶级 |
| tag | string, unique | 唯一标识契约 |
| created_at | timestamp | 创建时间 |
| updated_at | timestamp | 更新时间 |

### 3.2 menus 表

菜单发布池，存储前端可用的菜单项，保持树形结构。通过 parent_id 与 permissions 表保持一致的层级关系。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint, PK | 主键 |
| parent_id | bigint, nullable | 父级 id，null 为顶级（关联 permissions.id）|
| label | string | 菜单标题 |
| path | string | 路由路径（相对路径，动态拼接时使用）|
| component | string | 前端组件路径 |
| icon | string, nullable | 图标类名 |
| created_at | timestamp | 创建时间 |
| updated_at | timestamp | 更新时间 |

### 3.3 desktops 表

用户桌面配置。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint, PK | 主键 |
| user_id | bigint | 用户 id |
| name | string | 桌面名称 |
| region | string | 布局区域（如 `sidebar_left`, `header`）|
| created_at | timestamp | 创建时间 |
| updated_at | timestamp | 更新时间 |

### 3.4 desktop_menus 表

用户可将菜单从 menus 拖拽到桌面，并支持自定义覆盖原菜单项的值。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint, PK | 主键 |
| desktop_id | bigint, index | 关联 desktops.id |
| menu_id | bigint, index | 关联 menus.id |
| custom | json, nullable | 自定义覆盖值（如 `{"label": "我的文章", "path": "my-article"}`）|
| sort | int | 排序 |
| created_at | timestamp | 创建时间 |
| updated_at | timestamp | 更新时间 |

> 模型层自动合并 custom 与 menus 原值，优先使用 custom。

## 四、配置结构

模块通过 `composer.json` 的 `extra.platform-module` 声明权限和菜单：

```json
{
    "name": "third-party/module-article",
    "extra": {
        "platform-module": {
            "permissions": [
                {
                    "tag": "article",
                    "route": {
                        "component": "Article/List",
                        "icon": "fa-file"
                    },
                    "children": [
                        {
                            "tag": "add",
                            "route": {
                                "path": "add",
                                "component": "Article/Add"
                            }
                        },
                        {
                            "tag": "edit"
                        }
                    ]
                }
            ]
        }
    }
}
```

### 落库规则

| 规则 | 说明 |
|------|------|
| path 直接存储 | 仅存储原始 path，不做拼接处理 |
| 有 component 且无 path 时 | 自动生成 path = tag |
| 无 route 字段的节点 | 仅入库 permissions，不生成 menus 记录 |

### 数据返回规则

| 规则 | 说明 |
|------|------|
| path 动态拼接 | 返回菜单树时，相对路径自动拼接父级 path |
| 绝对路径 | 以 `/` 开头的不做拼接 |

### 落库结果

**permissions 表**

| id | parent_id | tag |
|----|-----------|-----|
| 1 | null | article |
| 2 | 1 | add |
| 3 | 1 | edit |

**menus 表**

| id | parent_id | label | path | component | icon |
|----|-----------|-------|------|-----------|------|
| 1 | null | article | article | Article/List | fa-file |
| 2 | 1 | add | add | Article/Add | null |

> 注：有 component 且无 path 时自动生成 `tag`。menus 通过 parent_id 关联 permissions 表层级。落库时 path 保持原始值，动态返回时拼接。

## 五、工作流程

```
┌─────────────────────────────────────────────────────────┐
│  模块安装                                                │
│    ↓                                                    │
│  解析 composer.json extra.platform-module               │
│    ↓                                                    │
│  落库 permissions 表（权限树结构）                        │
│  落库 menus 表（保持树结构，原样存储 path）                │
│    ↓                                                    │
│  管理员可见：所有 menus                                  │
│    ↓                                                    │
│  用户操作：从菜单池拖拽到个人桌面（desktop_menus）          │
│    ↓                                                    │
│  前端请求菜单 → 后端动态拼接 path → 返回完整菜单树          │
└─────────────────────────────────────────────────────────┘
```

## 六、命名约定

| 旧名称 | 新名称 | 原因 |
|--------|--------|------|
| routes | menus | 避免与后端路由冲突 |
| pid | parent_id | Laravel 生态惯例 |
| position | region | 更清晰的空间定义 |

## 七、多国语言

name 通过语言包匹配 tag 获取，不存储在数据库：

```json
{
    "zh_CN": {
        "article": "文章",
        "add": "新增",
        "edit": "编辑"
    },
    "en": {
        "article": "Article",
        "add": "Add",
        "edit": "Edit"
    }
}
```

---

## 八、设计优势

1. **零配置心智负担** - 只需声明 tag，其他自动推断
2. **权限与展示解耦** - permissions 控制权限，menus 控制展示
3. **用户高度自定义** - 每个用户可拥有独立的桌面布局
4. **模块化** - 第三方模块即装即用，自动注册
5. **Laravel 生态兼容** - 遵循 Laravel 命名惯例
6. **路径动态拼接** - 落库时保持简洁，渲染时动态拼接完整路径
7. **菜单树结构** - menus 与 permissions 保持一致的树形层级
