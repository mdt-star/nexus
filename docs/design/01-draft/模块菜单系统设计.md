# 模块菜单系统设计

## 一、设计背景

设计一个轻量化的模块菜单系统，支持第三方模块通过 JSON 配置声明权限，后端自动解析落库到 `permissions` 表。用户可从菜单池拖拽绑定到个人桌面，实现高度自定义的工作空间。

**权限与路由的关系：** 权限不再通过 `->middleware('auth.tag:...')` 手动绑定，而是由 `Route::auth()` 自动注入，中间件三层降级推断 tag。

---

## 二、核心设计理念

| 理念 | 说明 |
|------|------|
| 权限与展示分离 | permissions 负责权限树结构，menus 负责菜单展示数据 |
| 配置轻量化 | 自动推断字段，减少冗余配置 |
| 用户自定义 | menus 仅发布，不直接展示；用户从菜单池拖拽到桌面 |
| 多国语言 | name 通过语言包匹配 tag，数据库只存结构 |
| 零配置路由权限 | tag 从控制器和方法自动推断，无需显式声明 |

---

## 三、表结构设计

### 3.1 permissions 表

功能权限标记字典表，仅存储层级结构。
本表存储的是功能权限标记（Feature Flag），与 model_accesses（模型访问权限）无关。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint, PK | 主键 |
| parent_id | bigint, nullable | 父级 id，null 为顶级 |
| package_id | bigint, nullable | 关联 packages.id，null 为全局权限 |
| tag | string | 权限标识（如 article:list, article:add） |
| created_at | timestamp | 创建时间 |
| updated_at | timestamp | 更新时间 |

唯一性：`unique(['package_id', 'tag'])`，不同包可以有同名 tag。

### 3.2 menus 表

菜单发布池，存储前端可用的菜单项，保持树形结构。通过 parent_id 与 permissions 表保持一致的层级关系。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint, PK | 主键 |
| parent_id | bigint, nullable | 父级 id，null 为顶级（关联 permissions.id）|
| label | string | 菜单标题 |
| path | string | 路由路径（相对路径，动态拼接时使用）|
| component | string | 前端组件路径 |
| icon | string, nullable | 图标类名 |
| created_at | timestamp | 创建时间 |
| updated_at | timestamp | 更新时间 |

### 3.3 desktops 表

用户桌面配置。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint, PK | 主键 |
| user_id | bigint | 用户 id |
| name | string | 桌面名称 |
| region | string | 布局区域（如 `sidebar_left`, `header`）|
| created_at | timestamp | 创建时间 |
| updated_at | timestamp | 更新时间 |

### 3.4 desktop_menus 表

用户可将菜单从 menus 拖拽到桌面，并支持自定义覆盖原菜单项的值。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint, PK | 主键 |
| desktop_id | bigint, index | 关联 desktops.id |
| menu_id | bigint, index | 关联 menus.id |
| custom | json, nullable | 自定义覆盖值（如 `{"label": "我的文章", "path": "my-article"}`）|
| sort | int | 排序 |
| created_at | timestamp | 创建时间 |
| updated_at | timestamp | 更新时间 |

> 模型层自动合并 custom 与 menus 原值，优先使用 custom。

## 四、配置结构

模块通过 `composer.json` 的 `extra.nexus` 声明权限：

```json
{
    "name": "third-party/module-article",
    "extra": {
        "nexus": {
            "permissions": [
                {
                    "tag": "article",
                    "children": ["list", "add", "detail", "edit", "delete"]
                }
            ]
        }
    }
}
```

> 注意：permissions 配置中不再包含 menu 字段。权限只负责"能不能访问"，前端页面目录结构由前端自己维护。

### 落库规则

| 规则 | 说明 |
|------|------|
| 子节点 tag 拼接 | 自动拼接父级 tag，`article:list`、`article:add` |
| package_id 自动关联 | 同步时自动写入包 ID |
| 无 menu 字段 | permissions 配置中已取消 menu 字段 |

### 落库结果

**permissions 表**

| id | parent_id | package_id | tag |
|----|-----------|-----------|-----|
| 1 | null | 1 | article |
| 2 | 1 | 1 | article:list |
| 3 | 1 | 1 | article:add |
| 4 | 1 | 1 | article:detail |
| 5 | 1 | 1 | article:edit |
| 6 | 1 | 1 | article:delete |

## 五、路由权限系统

### 5.1 Route::auth() 宏

```php
Route::auth(function () {
    // 包名自动推断 — 通过 debug_backtrace 从调用文件路径提取
    Route::get('/articles', [ArticleController::class, 'index']);  // → article:list
    Route::post('/articles', [ArticleController::class, 'store']); // → article:add
});

// 显式指定包名
Route::auth('third-party/module-article', function () {
    Route::get('/articles', [ArticleController::class, 'index']);
});
```

`Route::auth()` 自动将 `package_id` 和 `package_name` 注入到路由 defaults 中，使中间件能精确查询权限。

### 5.2 Route::tag() 宏

```php
Route::auth(function () {
    Route::get('/custom', [CustomController::class, 'index'])->tag('custom:tag');
});
```

用于非标准路由，显式指定权限标识。

### 5.3 中间件三层降级推断

VerifyAuthTagMiddleware 的 tag 获取策略：

```
1. 中间件参数（如 auth.tag:article:list）
2. defaults('auth_tag')（来自 ->tag() 方法）
3. 自动推断 — 控制器名:方法映射
```

方法映射表：

| 控制器方法 | 推断的 tag | 说明 |
|-----------|-----------|------|
| index | {resource}:list | 列表 |
| create/store | {resource}:add | 新增 |
| show | {resource}:detail | 详情 |
| edit/update | {resource}:edit | 编辑 |
| destroy | {resource}:delete | 删除 |

推断规则：
- 控制器名去掉 `Controller` 后缀，首字母小写 → `article`
- 方法名按语义映射
- 组合为 `{控制器}:{动作}`

### 5.4 package_id 安全查询

| 条件 | 查询方式 |
|------|---------|
| 有 defaults('package_id') | `where('package_id', $id)->where('tag', $tag)` |
| 无 defaults('package_id') | `whereNull('package_id')->where('tag', $tag)` |

防止跨包权限误判。

## 六、三种路由权限使用方式对比

| 方式 | 代码 | package_id | tag 来源 | 适用场景 |
|------|------|-----------|---------|---------|
| 自动推断 | `Route::get('/articles', [Ctr::class, 'index'])` | 自动注入 | 控制器:方法映射 | 标准 RESTful 路由 |
| ->tag() | `->tag('custom:tag')` | 自动注入 | defaults('auth_tag') | 非标准路由 |
| 原生中间件 | `->middleware('auth.tag:tag')` | null | 中间件参数 | Route::auth() 外部 |

## 七、工作流程

```
┌─────────────────────────────────────────────────────────┐
│  模块安装                                                │
│    ↓                                                    │
│  解析 composer.json extra.nexus                         │
│    ↓                                                    │
│  落库 permissions 表（功能标记树结构）                     │
│    ↓                                                    │
│  路由定义 → Route::auth() 自动注入 package_id            │
│    ↓                                                    │
│  请求到达 → 中间件自动推断 tag → 校验权限                  │
│    ↓                                                    │
│  管理员可见：所有 menus                                  │
│    ↓                                                    │
│  用户操作：从菜单池拖拽到个人桌面（desktop_menus）          │
│    ↓                                                    │
│  前端请求菜单 → 后端动态拼接 path → 返回完整菜单树          │
└─────────────────────────────────────────────────────────┘
```

## 九、设计优势

1. **零配置路由权限** - tag 从控制器和方法自动推断，无需显式声明
2. **自动包关联** - Route::auth() 自动注入 package_id，精确查询
3. **三层降级兼容** - 支持自动推断、->tag()、原生中间件三种方式
4. **权限安全** - package_id + tag 复合唯一，杜绝跨包误判
5. **权限与展示解耦** - permissions 控制权限，menus 控制展示
6. **用户高度自定义** - 每个用户可拥有独立的桌面布局
7. **模块化** - 第三方模块即装即用，自动注册
8. **Laravel 生态兼容** - 遵循 Laravel 命名惯例，支持路由缓存
9. **路径动态拼接** - 落库时保持简洁，渲染时动态拼接完整路径
