# 项目架构设计

## 一、项目概述

`mdt-star/nexus` 是一个 Laravel 核心权限模块，提供功能权限控制、菜单管理、数据范围策略、动态配置等基础设施能力。

---

## 二、核心概念

### 两个核心权限概念

| 概念 | 说明 | 控制层面 | 生效范围 |
|------|------|---------|---------|
| **Permission** | 功能权限标记（Feature Flag），通过 tag 标识 | 路由/页面/元素 | 前后端请求层 |
| **ModelAccess** | 模型访问权限，控制主体对特定模型的读写删及数据范围 | Eloquent Model | 模型层（ORM） |

> **Permission** 控制"能不能看到/访问这个按钮或页面"；**ModelAccess** 控制"能不能查/改/删这类数据"。

### 其他概念

| 概念 | 说明 |
|------|------|
| Menu | 菜单发布池，前端可用的菜单项 |
| Desktop | 用户桌面，支持自定义布局 |
| DataScope | 数据范围策略，控制查询数据边界 |
| DynamicConfig | 动态配置，支持运行时修改 |
| Role | 用户组/角色，作为多态作用主体之一 |
| Subject | 作用主体（User/Role/Team 等），通过多态关联持有模型访问权限 |

## 六、核心类说明

### 6.1 NexusServiceProvider

核心服务提供者，负责：
- 注册模型观察者
- 注入全局 DataScope
- 加载动态配置（`mergePersistentConfig()` 自动合并数据库配置）
- 发布配置文件和语言文件
- 将本包 User 模型注入为 Laravel 默认认证用户模型（`registerUserProvider()`）
- 注册 Builder Macro（`setSubject`/`getSubject`/`hasSubject`）
- 自动检测包变更（`detectPackageChanges()`）
- 注册 Route::auth() 和 Route::tag() 路由宏（`registerRouteMacros()`）

### 6.2 PermissionSyncer

功能权限同步器，负责：
- 解析模块 composer.json 的 permissions 配置
- 落库 permissions 表（不再同步 menus）
- 子节点 tag 自动拼接为 `父tag:子tag`
- 直接通过 package_id 关联包（去掉多态关联）

### 6.3 MenuTreeBuilder

菜单树构建器，负责：
- 从 menus 表构建树形结构
- 动态拼接 path
- 合并 desktop_menus 的 custom 覆盖值

### 6.4 HasDataScope

ORM Trait，负责：
- 通过作用主体（Subject）的 `getModelAccess()` 方法消费模型访问权限
- 自动注入数据范围查询条件
- 读写删权限熔断（抛出 `PermissionDeniedException`）
- 支持 `setScopeSubject()` 注入其他作用主体
- 支持 `withoutDataScope()` 跳过作用域

### 6.5 VerifyAuthTagMiddleware

功能权限标记校验中间件，负责：
- 校验当前用户是否拥有指定功能权限 tag
- **三层降级推断策略**：中间件参数 → defaults('auth_tag') → 自动推断（控制器名:方法映射）
- 检查的是 permissions 表中的功能标记，与 model_accesses 无关
- package_id 安全查询（有 package_id 精确查，无则查全局）

### 6.6 HasModelAccess（契约接口）

模型访问权限主体契约，定义 `getModelAccess(?string $modelClass): Collection` 方法。
User、Role 等模型实现此接口，作为 model_accesses 的多态作用主体。

### 6.7 HasModelAccessTrait

模型访问权限主体通用 Trait，提供：
- `modelAccesses()` → `morphMany(ModelAccess::class, 'subject')` 多态关联
- `getModelAccess()` 默认实现（仅查自身），带缓存支持

### 6.8 User 模型

- 继承 `Illuminate\Foundation\Auth\User`（Authenticatable）
- 实现 `HasModelAccess` 接口
- `getModelAccess()` 穿透逻辑：自身权限 + 角色组权限，合并所有记录
- 通过 `NexusServiceProvider` 注入为 Laravel 默认认证用户模型

> **多角色权限合并策略**：读/写/删权限取并集（任一角色有权限即可），scope 策略叠加到查询中。
> 详见 [十二、多角色权限冲突与合并策略](#十二多角色权限冲突与合并策略)。

### 6.9 Role 模型

- 继承 `Illuminate\Database\Eloquent\Model`
- 实现 `HasModelAccess` 接口（使用 Trait 默认实现，仅查自身）
- 作为多态作用主体之一

### 6.10 PermissionDeniedException

- 继承 `Symfony\Component\HttpKernel\Exception\HttpException`（403）
- 消息通过 `nexus::exceptions.{key}` 语言包加载，支持国际化
- 所有权限拒绝场景统一抛出此异常

### 6.11 Filterable Trait

统一查询过滤 Trait，负责：
- 提供 `scopeFilter()` 作用域，接收结构化过滤条件
- 支持 13 种操作符：eq / neq / like / gt / gte / lt / lte / in / not_in / between / not_between / null / not_null
- 纯查询构建，不参与权限控制
- 配合 FilterRequest 使用

### 6.12 FilterRequest（抽象基类）

统一查询过滤请求基类，负责：
- 子类通过 `filterRules()` 声明式定义"能查什么字段、用什么操作符、默认值是什么"
- `filters()` 将请求参数解析为 Filterable 需要的结构化格式
- 支持默认值（请求未传参时自动应用）
- 自动根据 filterRules 生成 Laravel 验证规则
- `authorize()` 控制谁能访问

### 6.13 TypedValueCast

Laravel Custom Cast，负责：
- 根据 Config 模型的 type 字段动态转换 value 类型
- 支持 boolean / number / json / null / string
- 写入时自动推断类型并序列化

### 6.14 DynamicConfigManager

动态配置管理器，负责：
- 动态配置读写，支持运行时修改
- `mergeIntoConfig()` 将数据库配置合并到 Laravel Config Repository
- `set()` 写入后自动 `config()->set()` 即时生效
- `delete()` 删除后恢复静态配置值

## 七、权限体系

### 7.1 功能权限标记（Permission）

来自模块 composer.json 的 `extra.nexus.permissions` 配置，通过 tag 标识。
同一个 tag 可同时控制前端路由、后端路由、页面元素，实现"一次勾选，前后端都生效"。

权限与路由自动关联：
- `Route::auth()` 自动注入 package_id
- `VerifyAuthTagMiddleware` 三层降级推断 tag
- 支持自动推断、->tag()、原生中间件三种方式

### 7.2 模型访问权限（ModelAccess）

```
User (extends Authenticatable, implements HasModelAccess, use HasModelAccessTrait)
 ├── modelAccesses() → morphMany(ModelAccess::class, 'subject')
 ├── getModelAccess() → 自身权限 + 穿透 roles
 │     ├── $this->modelAccesses() 查自身
 │     └── 遍历 $this->roles → role->getModelAccess()
 └── roles() → BelongsToMany('role_user')

Role (extends Model, implements HasModelAccess, use HasModelAccessTrait)
 ├── modelAccesses() → morphMany(ModelAccess::class, 'subject')
 ├── getModelAccess() → 只查自身（Trait 默认实现）
 └── users() → BelongsToMany('role_user')

ModelAccess
 └── subject() → morphTo() 多态关联作用主体
```

## 十、路由权限系统

### 10.1 Route::auth() 宏

自动注入 package_id 到路由 defaults，支持自动推断和显式指定包名。

```php
// 自动推断（推荐，在模块 ServiceProvider 中使用）
Route::auth(function () {
    Route::get('/articles', [ArticleController::class, 'index']);
});

// 显式指定包名
Route::auth('third-party/module-article', function () {
    Route::get('/articles', [ArticleController::class, 'index']);
});
```

### 10.2 Route::tag() 宏

将自定义 tag 写入 defaults('auth_tag')，用于非标准路由。

```php
Route::auth(function () {
    Route::get('/custom', [CustomController::class, 'index'])->tag('custom:tag');
});
```

### 10.3 中间件三层降级

| 方式 | tag 来源 | package_id | 适用场景 |
|------|---------|-----------|---------|
| 自动推断 | 控制器名:方法映射 | 自动注入 | 标准 RESTful 路由 |
| ->tag() | defaults('auth_tag') | 自动注入 | 非标准路由 |
| 原生中间件 | 中间件参数 | null（查全局） | Route::auth() 外部 |

## 十一、路由挂载系统（Mount System）

### 11.1 概述

路由挂载系统（Mount System）提供了一种声明式、可继承的路由组织方式。通过预定义"挂载点"（Mount），开发者可以快速注册带有统一前缀和中间件能力的路由组。

核心概念：
- **Mount** = 前缀 + 一组底层能力（abilities）
- **Ability** = 可复用的路由配置管道（如 auth 中间件）
- **继承** = 一个 mount 可以继承一个或多个 mount，自动合并前缀和能力

### 11.2 预定义 Mount

Nexus 基础包预定义了以下 mount（继承链：admin → api → auth）：

| 名称 | 前缀 | 中间件 | defaults | 说明 |
|------|------|--------|----------|------|
| `auth` | `/api/{version}` | `auth.tag` | `package_id`, `package_name` | 基础认证+权限层，自动注入包信息 |
| `api` | `/api/{version}` | 继承 auth | 继承 auth | 继承 auth，可扩展 api 级中间件 |
| `admin` | `/api/{version}/admin` | 继承 api | 继承 api | 管理端路由，继承 api |

```php
// auth mount：基础认证+权限层
$manager->extend('auth', function () {
    $package = Package::findByComposer();  // 自动探测当前包
    return [
        'prefix' => '/api/{version}',
        'middlewares' => ['auth.tag'],
        'defaults' => [
            'package_id' => $package?->id,
            'package_name' => 'mdt-star/nexus',
        ],
    ];
});

// api mount：继承 auth
$manager->extend('api', function (string $version = 'v1') {
    return [
        'extends' => 'auth',
        'prefix' => "/api/{$version}",
    ];
});

// admin mount：继承 api
$manager->extend('admin', function (string $version = 'v1') {
    return [
        'extends' => "api:{$version}",
        'prefix' => '/admin',
    ];
});
```

**defaults 继承与读取：**
- mount 定义的 `defaults` 通过 `Route::group(['defaults' => [...]])` 注入给所有子路由
- 子 mount 的 defaults 覆盖父级同名 key
- `VerifyAuthTagMiddleware` 读取时同时检查 `$route->defaults` 和 `$route->action['defaults']`（兼容 Route::group 注入路径）
- `package_name` 通过 `Package::idByName()` 全表缓存 O(1) 解析

### 11.3 使用方式

**基础使用：**
```php
// 使用 mount 规格，回调接收 $route 参数（MountInstance 实例）
Route::mount('api', function ($route) {
    $route->get('/users', [UserController::class, 'index']);
});

// 指定版本
Route::mount('api:v2', function ($route) {
    $route->get('/users', [UserController::class, 'index']);
});
```

**快捷宏（自动注册）：**
```php
// 等价于 Route::mount('api', ...)，回调接收 $route 参数
Route::api(function ($route) {
    $route->get('/users', [UserController::class, 'index']);
});

// 带参数
Route::api('v2', function ($route) {
    $route->get('/users', [UserController::class, 'index']);
});
```

**取消能力：**
```php
// 取消 auth 能力（无需认证的路由），回调接收 $route 参数
Route::mount('api')->withoutAuth(function ($route) {
    $route->get('/public', [PublicController::class, 'index']);
});
```

**链式调用：**
```php
// withoutAuth() 后直接链式调用 get()，返回 Route 对象
Route::mount('api')->withoutAuth()->get('/public', function () {
    return 'public';
});
```

### 11.4 扩展自定义 Mount

第三方模块可以通过 `Route::extendMount()` 注册自己的 mount：

```php
// 在模块的 ServiceProvider 中
Route::extendMount('admin', function (string $version = 'v1') {
    return [
        'extends' => "api:{$version}",
        'prefix' => '/admin',
    ];
});

// 使用
Route::mount('admin', function () {
    Route::get('/dashboard', [DashboardController::class, 'index']);
});
// 等价于：/api/v1/admin/dashboard，带 auth 中间件
```

### 11.5 继承机制

Mount 支持单继承和多继承：

```php
// 单继承
Route::extendMount('child', function () {
    return [
        'extends' => 'parent',
        'prefix' => '/child',
    ];
});

// 多继承
Route::extendMount('child', function () {
    return [
        'extends' => ['parent1', 'parent2'],
        'prefix' => '/child',
    ];
});
```

继承规则：
- **前缀合并**：子 mount 的前缀始终追加到父级后面
- **能力合并**：子 mount 继承父级的所有能力，子级可覆盖
- **参数传递**：通过 `extends:parent:v2` 语法向父级传递参数

### 11.6 扩展能力

```php
// 注册自定义能力
Route::extendAbility('audit', function ($route) {
    return $route->middleware('audit.log');
});

// 在 mount 中使用
Route::extendMount('secure', function () {
    return [
        'extends' => 'api',
        'prefix' => '/secure',
        'abilities' => ['auth', 'audit'],
    ];
});
```

### 11.7 MountManager 核心类

`MountManager` 是挂载系统的核心，负责：
- `extend(string $name, callable $resolver)` - 注册 mount 定义
- `extendAbility(string $name, callable $handler)` - 注册能力定义
- `mount(string $spec, callable $callback)` - 解析并执行 mount
- `instance(string $spec)` - 创建 MountInstance（支持链式取消能力）

### 11.8 MountInstance 链式调用

`MountInstance` 支持链式取消能力：

```php
Route::mount('api')
    ->withoutAuth()
    ->withoutAudit()
    ->group(function () {
        Route::get('/public', [PublicController::class, 'index']);
    });
```

每个 `without{Ability}()` 方法对应一个能力名称，自动将能力名转为驼峰格式。

## 十二、超级管理员

### 12.1 概述

超级管理员（Super Admin）拥有至高无上的权限，可以跳过所有权限检查。默认为系统中第一个用户（id = 1），可通过配置自定义。

### 12.2 配置

```php
// config/nexus.php
'super_admin_id' => env('NEXUS_SUPER_ADMIN_ID', 1),
```

### 12.3 影响范围

超级管理员在以下场景中跳过权限检查：

| 组件 | 行为 |
|------|------|
| `User::isSuperAdmin()` | 返回 true，跳过所有权限检查 |
| `VerifyAuthTagMiddleware` | 超级管理员直接放行，不校验功能权限 tag |
| `HasDataScope` | 超级管理员跳过数据范围过滤，可查看所有数据 |

### 12.4 实现

```php
public function isSuperAdmin(): bool
{
    $superAdminId = config('nexus.super_admin_id', 1);
    return $this->id === (int) $superAdminId;
}
```

## 十三、设计原则

1. **零物理外键** - 逻辑关联，表结构独立
2. **Laravel 生态兼容** - 遵循命名规范，使用生态工具，支持路由缓存
3. **配置驱动** - 静态配置放 config，动态配置放数据库
4. **模块化** - 第三方模块即装即用
5. **单一职责** - 每个类只做一件事
6. **多态作用主体** - User/Role/Team 等通过多态关联持有模型访问权限
7. **国际化异常** - 权限拒绝异常通过语言包支持多语言
8. **权限概念分离** - Permission（功能标记）与 ModelAccess（模型访问权限）严格区分
9. **声明式过滤** - FilterRequest 定义规则，Filterable 执行查询，Model 不参与
10. **零配置路由权限** - tag 从控制器和方法自动推断，无需手动声明
11. **权限安全** - package_id + tag 复合唯一，杜绝跨包误判

## 十四、多角色权限冲突与合并策略

### 14.1 问题背景

用户可同时属于多个角色组（Role），每个角色组可独立配置模型访问权限（ModelAccess）。
当多个角色对同一模型类有不同的权限设置时，需要明确的优先级规则来决定最终生效的权限。

### 14.2 合并原则

**严格优先级，高优先级覆盖低优先级。**

```
用户自身权限（最高优先级）
    ↓ 覆盖（按 class 匹配）
第一个角色（主角色，次高优先级）
    ↓ 覆盖（按 class 匹配）
第二个角色（第三优先级）
    ↓ 覆盖（按 class 匹配）
第三个角色（第四优先级）
    ↓ ...
```

### 14.3 排序机制

`role_user` 中间表增加 `sort` 字段（unsignedSmallInteger，默认 0），用于控制角色优先级。sort 越小优先级越高。

`User::roles()` 关联按 `pivot_sort` 升序排列，确保第一个返回的角色为主角色。

用户在前端拖拽排序后，调用 API 更新 `role_user.sort` 即可。

### 14.4 合并逻辑

`User::getModelAccess()` 的合并流程：

1. 查询用户自身权限（最高优先级）
2. 按 sort 升序遍历角色组，高优先级角色覆盖低优先级角色的相同 class 记录
3. 用户自身权限覆盖角色权限（相同 class 时以用户为准）

### 14.5 执行流程示例

用户有角色 A（主角色，sort=0）、角色 B（第二角色，sort=1），对 Article 模型：

| 来源 | can_read | can_write | scope_key |
|------|----------|-----------|-----------|
| 角色 B（第二角色） | true | false | org_only |
| 角色 A（主角色） | false | true | self_only |
| 用户自身 | true | true | null |

合并过程：
1. 先处理角色 A（主角色），其 Article 记录进入结果集
2. 再处理角色 B，其 Article 记录与角色 A 冲突（相同 class），被剔除
3. 用户自身权限覆盖角色权限
4. 最终结果：can_read=true, can_write=true, scope_key=null

### 14.6 HasDataScope 中的处理

遍历最终合并后的 access 记录：
- can_read=false → 抛异常（最高优先级主体明确拒绝）
- can_write=false → 抛异常
- can_delete=false → 抛异常
- scope_key → 应用策略

因为已经按优先级合并好了，所以遍历时遇到 false 直接抛异常是正确的——这代表最高优先级的主体明确拒绝了该操作。

### 14.7 影响范围

| 文件 | 改动 |
|------|------|
| role_user 迁移文件 | 增加 sort 字段 |
| User 模型 | roles() 加排序，getModelAccess() 加角色间覆盖 |
| Role 模型 | 无改动 |
